How Supabase Auth + RLS Work Together

Three concepts from the supabase-react-crud experiment, explained interactively

Every Supabase request is a four-step chain

When your React app reads or writes data, the request passes through four layers. The key insight: your database knows who the user is because the JWT travels all the way down to Postgres.

Step 0 / 7
Sign in + Fetch notes
1
Auth
User signs in with email + password
supabase.auth.signInWithPassword({ email, password })
↓
2
Auth service
Supabase Auth verifies credentials, returns a JWT
The JWT contains the user's UUID. Your app stores it automatically.
↓
3
Client
App calls supabase.from('notes').select('*')
The supabase-js client automatically attaches the JWT to the request header.
↓
4
REST API (PostgREST)
API receives the request, extracts the user ID from the JWT
Sets the Postgres session variable so auth.uid() returns this user's ID.
↓
5
Postgres
Runs SELECT * FROM notes — but RLS policy kicks in
Policy says: only return rows WHERE auth.uid() = user_id. Postgres filters before sending results.
↓
6
REST API
Returns only the rows that passed the RLS check
The API doesn't filter — it just forwards what Postgres gave it.
↓
7
Client
React component receives only this user's notes
No client-side filtering needed. The database handled it.

Key takeaway: Your React code never filters by user. It just says select('*'). The security lives in Postgres, not in your JavaScript. Even if someone tampered with the client code, they couldn't see other users' data — the database won't return it.

Row-Level Security filters at the database level

The notes table contains everyone's data. But each user only ever sees their own rows. Toggle between users to see what the same select('*') query returns for each.

What's actually in the database
id user_id title body
a1b2...user-a-uuidFirst note from User AThis should only be visible to User A
c3d4...user-a-uuidSecond note from User AAlso private to A
e5f6...user-b-uuidUser B's noteOnly B can see this
Viewing as:
-- Your React code runs this:
SELECT * FROM notes;

-- Postgres silently appends the RLS policy:
SELECT * FROM notes WHERE auth.uid() = user_id;
-- auth.uid() = 'user-a-uuid'
Result: 2 rows returned (User A's notes only)

Key takeaway: RLS means you write simple queries (select *, insert, delete) and the database enforces the rules. You don't need to remember to add .eq('user_id', currentUser.id) to every query — and more importantly, you can't forget to, because the policy is always active.

One SQL file defines your entire backend

The migration file is the source of truth — the table schema, security settings, and access rules. Hover over each section to understand what it does.

Key takeaway: This single SQL file replaces what would otherwise be dashboard clicks for creating a table, toggling RLS on, and writing policies. It's version-controlled, reviewable in PRs, and reproducible — run supabase db reset on any machine and you get the same database.
