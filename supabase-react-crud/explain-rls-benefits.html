Why RLS Matters

The same app, two approaches — and what happens when things go wrong

The same feature, two ways

Both approaches fetch the current user's notes. Look at what the developer has to remember in each case.

Fetching notes

Without RLS — traditional approach
// Must filter by user on EVERY query const { data } = await supabase .from('notes') .select('*') .eq('user_id', user.id) // forget this = data leak .order('created_at')
With RLS
// Just ask for what you need const { data } = await supabase .from('notes') .select('*') // database filters automatically .order('created_at')

Deleting a note

Without RLS
// Must verify ownership manually const { data: note } = await supabase .from('notes') .select('user_id') .eq('id', noteId) .single() if (note.user_id !== user.id) { throw new Error('Not yours') } await supabase.from('notes') .delete().eq('id', noteId)
With RLS
// Just delete — policy blocks if not yours await supabase .from('notes') .delete() .eq('id', noteId) // policy: auth.uid() = user_id // wrong user? 0 rows affected. silent, safe.

Inserting a note

Without RLS
// Must set user_id from client await supabase.from('notes').insert({ title, body, user_id: user.id // client controls this! })

A malicious client could send any user_id here — inserting notes into another user's account.

With RLS
// user_id filled by database (default auth.uid()) await supabase.from('notes').insert({ title, body, // no user_id — Postgres fills from JWT })

Even if a client sends a fake user_id, the with check policy rejects it — auth.uid() must equal the row's user_id.

What happens when someone forgets a filter?

In any real project, someone will eventually write a query and forget to filter by user. A new team member, a rushed feature, a refactor that drops a line. Click through to see the consequences in each approach.

Scenario: A developer writes a new "search notes" feature

Without RLS — the bug
// New search feature — developer forgot the filter const { data } = await supabase .from('notes') .select('*') .ilike('title', `%${query}%`) // oops — no .eq('user_id', user.id)
API returns:
Click "Run search" to see results
With RLS — the same bug
// Same code — also "forgot" a filter const { data } = await supabase .from('notes') .select('*') .ilike('title', `%${query}%`) // no filter needed — RLS handles it
API returns:
Click "Run search" to see results

Without RLS: User A sees everyone's notes that match the search. This is a data breach. The bug might not be caught in testing if the developer only has one test account.

With RLS: The exact same "buggy" code is perfectly safe. Postgres appends WHERE auth.uid() = user_id to every query, regardless of what the client sends. The developer can't accidentally expose data because the safety net is in the database, not in their code.

The four benefits of RLS

1. You can't forget

Without RLS, security is opt-in — every query must manually filter. With RLS, security is opt-out — you'd have to explicitly bypass it (which requires the service role key, not the public one). The safe path is the default path.

2. Defense in depth

Even if your API has a bug, even if someone finds a way to call your REST endpoint directly, even if a developer skips the middleware — the database still enforces the rules. RLS is your last line of defense, and it never sleeps.

3. Simpler client code

Your queries say what you want, not who's allowed to see it. select('*') instead of select('*').eq('user_id', user.id). Less boilerplate, fewer bugs, easier to read. The access rules live where they belong — next to the data.

4. Works for every access path

React web app, mobile app, curl from the terminal, a new microservice you build next year — they all go through Postgres, so they all get the same rules. You define the policy once, and it applies everywhere, automatically.

Mental model: Think of RLS like a bouncer at the database door. Without RLS, every room in your app needs its own bouncer, and you're hiring them manually. With RLS, there's one bouncer at the only entrance, and nobody gets past without showing ID.
