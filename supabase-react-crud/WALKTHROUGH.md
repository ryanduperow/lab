# Walkthrough: supabase-react-crud

Read the files in this order. Infrastructure first, then client code.

---

## 1. `supabase/config.toml`

Generated by `supabase init`. Configures the local Supabase stack — ports, JWT secret, auth settings. The key default for this experiment: `enable_confirmations = false` under `[auth]`, which means `signUp` immediately creates a confirmed user (no email verification in local dev).

## 2. `supabase/migrations/00000000000000_create_notes.sql`

The single migration file — this is the core of "code-first Supabase."

- Creates a `notes` table with `id`, `user_id`, `title`, `body`, `created_at`
- `user_id` has `default auth.uid()` — Postgres fills this from the JWT automatically, so the client never sends a user ID
- Enables Row-Level Security on the table
- Defines three policies: SELECT, INSERT, DELETE — each scoped to `auth.uid() = user_id`

When you run `npx supabase db reset`, this file executes against the local Postgres instance.

## 3. `.env.local.example`

Template for environment variables. The URL is always `http://127.0.0.1:54321` for the local stack. The anon key comes from the `npx supabase start` output — it's a fixed JWT derived from the secret in `config.toml`.

Vite requires the `VITE_` prefix to expose env vars to client code. Accessed via `import.meta.env.VITE_*`.

## 4. `src/supabaseClient.js`

Creates a single Supabase client instance using the URL and anon key from environment variables. Every component imports from this file — one client, one connection.

## 5. `src/main.jsx`

Standard Vite entry point. Renders `<App />` into the root div. Imports `index.css` for base styles.

## 6. `src/App.jsx`

The top-level orchestrator. Two responsibilities:

1. **Session management**: On mount, calls `supabase.auth.getSession()` to check for an existing session, then subscribes to `onAuthStateChange` for real-time auth state updates (sign in, sign out, token refresh).
2. **Conditional rendering**: No session → show `<Auth />`. Has session → show the header (email + sign out button), `<NoteForm />`, and `<NoteList />`.

Uses a `refreshKey` counter to trigger `NoteList` re-fetches after adding a note.

## 7. `src/components/Auth.jsx`

Email + password form with two actions:

- **Sign in**: `supabase.auth.signInWithPassword({ email, password })` — returns an error if the user doesn't exist or the password is wrong.
- **Sign up**: `supabase.auth.signUp({ email, password })` — creates a new user in Supabase Auth. Locally, the user is auto-confirmed.

On success, `App.jsx` catches the new session via `onAuthStateChange` and re-renders.

## 8. `src/components/NoteForm.jsx`

A form with title and body inputs. On submit:

```js
await supabase.from('notes').insert({ title, body })
```

Notice: no `user_id` in the insert. The database column default (`auth.uid()`) fills it from the JWT. This is the cleanest ownership pattern — the client literally cannot forge ownership.

After a successful insert, calls `onNoteAdded()` to bump the refresh counter in `App.jsx`.

## 9. `src/components/NoteList.jsx`

Fetches notes on mount and whenever `refreshKey` changes:

```js
await supabase.from('notes').select('*').order('created_at', { ascending: false })
```

There's no `.eq('user_id', ...)` filter — **RLS does the filtering at the database level**. The query returns all notes, but Postgres only sends back rows where `auth.uid() = user_id`. This is the key insight: access control lives in the database, not in your application code.

Each note has a delete button that calls:

```js
await supabase.from('notes').delete().eq('id', id)
```

The DELETE policy ensures you can only delete your own notes — even if you somehow passed another user's note ID, the policy would reject it.
