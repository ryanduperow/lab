Harness Engineering

Reliability through scaffolding, not prompting — as demonstrated by the meal-planner-harness

📝 Skill (Prompt-based)

  • Instructions baked into a markdown file
  • AI decides whether to follow each step
  • Can skip steps, hallucinate, or quit early
  • Output format varies between runs
  • Entire workflow in one context window
  • No programmatic validation

⚙️ Harness (Code-controlled)

  • Workflow codified in deterministic phases
  • Code controls sequencing — AI can't skip steps
  • Each LLM call scoped to a narrow task
  • Output format guaranteed by templates
  • Sub-agents with isolated context windows
  • Programmatic validation at each gate

Phase 3 spawns one claude -p subprocess per day. Each runs in its own process with isolated context — no shared state, no context pollution. The harness assembles results afterward.

Each subprocess runs claude -p with --no-session-persistence — a completely fresh context window. The harness provides only what each sub-agent needs: the ingredient list, dietary prefs, and a JSON schema that constrains the output shape.

Phase 2 uses a callWithRetry wrapper. The harness validates LLM output programmatically and retries once if items are missing.

That's it. Two narrow, well-scoped tasks where the LLM excels — classification and creative generation. Everything else is deterministic code.
The harness principle: LLMs are unreliable at multi-step procedures but excellent at narrow tasks. Break the workflow into phases where code handles orchestration and the LLM only fills in what code can't do.
